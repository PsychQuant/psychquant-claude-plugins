### Distribution
- **macOS 26 codesigning + notarization**: release binary is now signed with Developer ID Application + hardened runtime + notarized via `xcrun notarytool`. macOS 26 tightened TCC such that ad-hoc signed binaries can no longer trigger Calendar / Reminders / AppleEvents permission dialogs — Developer ID + hardened runtime + notarization is the only path that lets end users grant access without manually re-signing the binary themselves.
  - New `scripts/sign-and-notarize.sh` wraps the codesign + ditto-zip + notarytool submit flow with pre-flight checks (cert in keychain, notarytool keychain profile configured) and friendly error messages including the submission ID for `xcrun notarytool log` post-mortem.
  - New `scripts/build-mcpb.sh` (or build-cli.sh) step `[3.5/N]` calls the signing script automatically. Auto-skips when `DEVELOPER_ID` env var is unset OR cert isn't in the keychain (so contributors / CI / forks can build a working unsigned artifact for testing without manually setting `SKIP_CODESIGN=1`).
  - New `Makefile` target `release-signed` is the canonical release-cut command; uses `REQUIRE_CODESIGN=1` to fail-fast if prerequisites missing.
  - `Sources/{{TARGET_NAME}}/Entitlements.plist` (empty `<dict/>`) — minimal entitlements; hardened runtime alone is what macOS 26 requires for TCC. Add specific entitlement keys here only if hardened runtime breaks a specific API at runtime.
  - **Known limitation**: stapling skipped (raw Mach-O doesn't support `xcrun stapler staple`); Gatekeeper online-checks notarization on first launch, requiring one-time network access.
  - **Upgrade trap fix**: `Makefile install:` and the user-facing install instructions both use `rm -f` before write, to prevent the inode-cache code-signature trap when upgrading over a previous install.
  - README "Release Process" gains a "Signing & Notarization" subsection with prerequisites, per-release flow, end-to-end verification (`spctl -a -vvv -t install`), and troubleshooting.
